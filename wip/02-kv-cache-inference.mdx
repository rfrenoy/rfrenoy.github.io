In the [previous post](/blog/01-gpt-model/), we built a GPT model in MLX that can do a forward pass and produce logits. We also gave it a `generate_naive` method that produces text — one token at a time, feeding the growing sequence back through the entire model each step.

It works. But it's spectacularly wasteful.

In this post, we'll understand *why* it's wasteful, build the data structure that fixes it (the **KV cache**), and wire it into an **inference engine** that can stream text efficiently. Along the way, we'll see one of the sharpest differences between PyTorch and MLX: how each framework handles mutable state.

## The problem with naive generation

Let's look at what happens when our model generates 5 tokens from a 3-token prompt:

```
Step 1:  model([A, B, C])           → predict token D     (3 tokens processed)
Step 2:  model([A, B, C, D])        → predict token E     (4 tokens processed)
Step 3:  model([A, B, C, D, E])     → predict token F     (5 tokens processed)
Step 4:  model([A, B, C, D, E, F])  → predict token G     (6 tokens processed)
Step 5:  model([A, B, C, D, E, F, G]) → predict token H   (7 tokens processed)
                                                     Total: 25 tokens processed
```

At each step, the model reprocesses the *entire* sequence from scratch. To generate $N$ tokens from a prompt of length $P$, we process a total of:

$$
\sum_{t=P}^{P+N-1} t = \frac{N(2P + N - 1)}{2} = O(N^2 + NP)
$$

tokens through the model. The cost is **quadratic** in the number of generated tokens. For a 1000-token prompt generating 1000 tokens, that's roughly a million forward passes through each layer, when only 1000 were necessary.

## The insight: attention reuses old computations

To see what's being wasted, let's look inside attention. At step 3, when we feed in `[A, B, C, D, E]`, the attention layer computes:

$$
Q = X W_Q, \quad K = X W_K, \quad V = X W_V
$$

then computes $\text{softmax}(QK^T) \cdot V$. The queries, keys, and values for tokens A, B, C, D are *exactly the same* as they were in previous steps — same input, same weights, same result. We're recomputing them for nothing.

Here's the key observation: **in causal attention, the output at position $t$ only depends on keys and values at positions $\leq t$**. So the K and V vectors for all previous tokens never change. If we store them, we can skip the recomputation entirely.

This is the KV cache.

## How the KV cache works

The idea is simple: maintain a running buffer of all previously computed K and V vectors. At each generation step, we only compute Q, K, V for the *new* token, append the new K and V to the cache, then run attention with the full history.

The generation now looks like:

```
Prefill:  model([A, B, C])    → cache K,V for A,B,C → predict D   (3 tokens)
Step 1:   model([D])           → cache K,V for D     → predict E   (1 token)
Step 2:   model([E])           → cache K,V for E     → predict F   (1 token)
Step 3:   model([F])           → cache K,V for F     → predict G   (1 token)
Step 4:   model([G])           → cache K,V for G     → predict H   (1 token)
                                                           Total: 7 tokens
```

That's 7 tokens instead of 25 — and in general $P + N$ instead of $O(N^2 + NP)$. **Linear instead of quadratic.** For the 1000+1000 example, that's 2000 instead of ~1,000,000.

There's one subtlety: the **attention computation** itself still involves the full cached sequence. When generating the $t$-th token, the new query (length 1) attends to all $t$ cached keys. So the attention cost per step is $O(t)$, and the total across all steps is $O(N^2)$ in attention alone. But this is much cheaper than recomputing *all* the projections, MLPs, and norms for every previous token.

## The cache data structure

The KV cache stores, for each layer, a pair of tensors that grow as we generate:

$$
K_{\text{cache}}^{(\ell)} \in \mathbb{R}^{B \times H_{kv} \times T_{\max} \times d}, \quad V_{\text{cache}}^{(\ell)} \in \mathbb{R}^{B \times H_{kv} \times T_{\max} \times d}
$$

where $B$ is batch size, $H_{kv}$ is the number of key/value heads (which may be less than query heads in GQA), $T_{\max}$ is the pre-allocated maximum sequence length, and $d$ is the head dimension.

We pre-allocate the full buffer up front and track the current fill position with a simple integer counter `pos`. At each step, we write new K, V into positions `[pos, pos + T_new)` and return the slice `[0, pos + T_new)` for attention.

Two differences stand out:

1. **Tensor layout.** nanochat stores `(B, T, H, D)` to match Flash Attention 3. We store `(B, H, T, D)` to match MLX's `scaled_dot_product_attention`.

2. **Position tracking.** nanochat uses a per-batch-element `cache_seqlens` tensor (FA3 needs it as a CUDA tensor for its kernel). We use a plain Python integer — simpler, and sufficient because all batch elements always advance in lockstep.

## The update operation: mutable vs. immutable

Here's where the frameworks diverge most. In PyTorch, Flash Attention 3 has a special API — `flash_attn_with_kvcache` — that updates the cache tensors **in place** during the attention computation. One fused kernel handles reading old K/V, appending new K/V, and computing attention. Elegant and fast.

MLX arrays are **immutable** — there's no in-place mutation. So we have to do the update ourselves:

    v=v,       # new values to append

    causal=True,

  rightCode={`# Step 1: Write new K, V into cache, get full history

    end = self.pos + k_new.shape[2]
    self.k_cache[layer_idx][:, :, self.pos:end, :] = k_new
    self.v_cache[layer_idx][:, :, self.pos:end, :] = v_new
    return (self.k_cache[layer_idx][:, :, :end, :],

The MLX version is more explicit — we manually write into the buffer and pass the full history to SDPA. In PyTorch, FA3 hides all of this behind one kernel call. The trade-off: our code is more transparent (you see exactly what's being cached and what's being computed), at the cost of potentially less fusion.

## Masks change with position

There's a subtle issue when using a KV cache: the attention mask shape changes.

During training (no cache), Q, K, and V all have shape `(B, H, T, D)` for the same T. The causal mask is a `(T, T)` lower-triangular matrix. But during cached generation, Q has length $T_{\text{new}}$ (usually 1 — just the new token) while K and V have length $T_{\text{cached}} + T_{\text{new}}$ (the full history). The mask must be `(T_new, T_cached + T_new)`.

For a single decode step where we're generating the token at absolute position $p$, the mask is a single row: `[True, True, ..., True]` — the new token can attend to all $p$ previous positions and itself. For the `"causal"` string shortcut in MLX (which assumes square Q, K lengths), we need to fall back to an explicit mask whenever there's a position offset:

```python
def _build_mask(self, T, window_size, offset=0):
    K_len = offset + T

    if window_size < 0 and offset == 0:
        return "causal"  # fast path: square mask, no cache

    # Query positions: [offset, offset+T)
    # Key positions:   [0, K_len)
    row = mx.arange(offset, offset + T).reshape(T, 1)
    col = mx.arange(K_len).reshape(1, K_len)

    mask = col <= row  # causal constraint
    if window_size >= 0:
        mask = mask & ((row - col) <= window_size)  # sliding window

    return mask
```

Think about what this means concretely. At position 50 with a window size of 10, the new token can attend to positions 40–50. The mask is a single row with 51 entries, where positions 0–39 are `False` and positions 40–50 are `True`.

## Prefill and decode: two phases of generation

Efficient generation has two distinct phases:

**Prefill** processes the entire prompt in one forward pass. All prompt tokens flow through the model in parallel (just like training), and their K, V vectors are stored in the cache. This is compute-bound — many tokens, lots of matrix multiplications.

**Decode** generates one token at a time. Each step processes a single new token, extending the cache by one position. This is memory-bound — the bottleneck is reading the full KV cache from memory, not the actual computation.

```python
# 1) Prefill — full prompt, batch_size=1
kv_cache = KVCache(batch_size=1, max_seq_len=prompt_len + max_tokens, ...)
ids = mx.array([prompt_tokens])            # (1, prompt_len)
logits = model(ids, kv_cache=kv_cache)     # fills cache
logits = logits[:, -1, :]                  # last position's prediction

# 2) Decode — one token at a time
for _ in range(max_tokens):
    next_id = sample(logits)               # (1, 1)
    logits = model(next_id, kv_cache=kv_cache)[:, -1, :]
    yield next_id
```

This two-phase pattern is universal across all LLM inference systems. The difference between a good and bad inference stack is mostly about how efficiently each phase is implemented.

## Multi-sample generation: clone the cache

Sometimes you want multiple completions from the same prompt (e.g., for best-of-N sampling in RL, or just to give the user variety). The naive approach is to prefill the prompt $N$ times. But the prompt is the same across all samples — its K, V are identical.

The trick: prefill once with batch_size=1, then **clone** the cache for each sample:

<CodeCompare

logits = logits[:, -1, :].expand(num_samples, -1)

# Clone for multi-sample decode
kv_cache_decode = KVCache(batch_size=num_samples, ...)
kv_cache_decode.prefill(kv_cache_prefill)
# Now decode with batch_size=num_samples`}
  rightTitle="nanochat-mlx (MLX)"
  rightCode={`# Prefill batch_size=1
kv_prefill = KVCache(batch_size=1, ...)
logits = model(ids, kv_cache=kv_prefill)
logits = logits[:, -1, :]

# Clone for multi-sample decode
logits = mx.broadcast_to(logits, (num_samples, V))
kv_decode = kv_prefill.clone_for_batch(num_samples)
mx.eval(kv_decode.k_cache, kv_decode.v_cache)
# Now decode with batch_size=num_samples`}
/>

The clone operation broadcasts each layer's `(1, H, T, D)` buffer to `(N, H, T, D)`. This costs $O(N \cdot T \cdot H \cdot d)$ memory, but saves re-running the entire model $N$ times on the prompt.

Note the `mx.eval(...)` call in the MLX version — this is MLX's lazy evaluation at work. Without it, the broadcast is recorded but not computed, and the prefill cache might be garbage-collected before the values are materialized. Explicit `eval` forces the computation to happen right now.

## The Engine: streaming with tool use

The `Engine` class wraps the model and tokenizer into a complete inference pipeline. Its `generate` method is a Python generator that yields one token column per step — perfect for streaming output to a terminal or web UI.

But there's a twist: nanochat's models can use **tools**. When the model outputs a `<|python_start|>` token, the engine intercepts the following tokens as a Python expression, evaluates it safely, and injects the result back into the token stream:

```
Model generates:  ... The answer is <|python_start|> 42 * 17 <|python_end|>
Engine intercepts: evaluates "42 * 17" = 714
Engine injects:    <|output_start|> 714 <|output_end|>
Model continues:   ... 714. So the total is ...
```

This is implemented as a simple state machine. Each batch row tracks whether it's currently inside a Python block, and accumulates expression tokens. When the block closes, the expression is evaluated with a sandboxed `eval` (no builtins, timeout protection) and the result tokens are pushed into a "forced tokens" queue that overrides sampling at the next steps.

```python
class RowState:
    current_tokens: list[int]       # full sequence so far
    forced_tokens: deque[int]       # tokens to force-inject next
    in_python_block: bool           # inside <|python_start|> ... <|python_end|>?
    python_expr_tokens: list[int]   # accumulator for the expression
    completed: bool                 # hit <|assistant_end|> or <|bos|>?
```

This design is identical between nanochat and nanochat-mlx — the tool-use logic is pure Python, independent of the tensor framework.

## The sampling step

Given logits of shape `(B, V)` — one score per vocabulary token per batch element — we need to pick the next token. The standard approach:

1. **Top-k filtering**: zero out everything below the $k$-th highest logit
2. **Temperature scaling**: divide logits by $T$ (lower = sharper, higher = more random)
3. **Categorical sampling**: convert to probabilities via softmax, then sample

$$
p_i = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}}
$$

<CodeCompare
  leftTitle="nanochat (PyTorch)"
  leftCode={`@torch.inference_mode()
def sample_next_token(logits, rng, temperature=1.0, top_k=None):
    if temperature == 0.0:
        return torch.argmax(logits, dim=-1, keepdim=True)
    if top_k is not None and top_k > 0:
        k = min(top_k, logits.size(-1))
        vals, idx = torch.topk(logits, k, dim=-1)
        vals = vals / temperature
        probs = F.softmax(vals, dim=-1)
        choice = torch.multinomial(probs, num_samples=1, generator=rng)
        return idx.gather(1, choice)
    else:
        logits = logits / temperature
        probs = F.softmax(logits, dim=-1)
        return torch.multinomial(probs, num_samples=1, generator=rng)`}
  rightTitle="nanochat-mlx (MLX)"
  rightCode={`def sample_next_token(logits, temperature=1.0, top_k=None):
    if temperature == 0.0:
        return mx.argmax(logits, axis=-1, keepdims=True)
    if top_k is not None and top_k > 0:
        k = min(top_k, logits.shape[-1])
        top_values = mx.topk(logits, k=k)
        threshold = top_values[:, -1:]
        logits = mx.where(logits < threshold, float("-inf"), logits)
    logits = logits / temperature
    next_ids = mx.random.categorical(logits)
    return next_ids.reshape(-1, 1)`}
/>

Three differences:

1. **`@torch.inference_mode()`** — PyTorch needs this decorator to disable autograd during inference. MLX doesn't build a gradient tape unless you explicitly call `mx.grad`, so no decorator needed.

2. **Random generator** — PyTorch passes an explicit `rng` generator for reproducibility. MLX uses a global seed set once with `mx.random.seed(42)`.

3. **Top-k implementation** — PyTorch's `torch.topk` returns both values and indices, so it can work in the reduced $k$-dimensional space (softmax over just the top-k, then gather the original indices). MLX's `mx.topk` returns only values, so we use a masking approach: find the threshold, set everything below it to $-\infty$, then sample from the full vocabulary (the $-\infty$ logits become zero probability after softmax).

## Proving correctness: the equivalence test

How do we know the KV cache produces the right answer? We wrote a test that generates tokens **both ways** — with and without cache — using greedy decoding (temperature=0, deterministic), and asserts they match:

```python
def test_equivalence_greedy(self, small_model, small_config):
    prompt = list(range(8))
    n_generate = 12

    # --- No cache: recompute full sequence each step ---
    ids = mx.array([prompt])
    tokens_no_cache = []
    for _ in range(n_generate):
        logits = small_model(ids)
        next_id = mx.argmax(logits[:, -1, :], axis=-1, keepdims=True)
        tokens_no_cache.append(next_id.item())
        ids = mx.concatenate([ids, next_id], axis=1)

    # --- With cache: prefill + single-token decode ---
    cache = KVCache(batch_size=1, ...)
    logits_prefill = small_model(mx.array([prompt]), kv_cache=cache)
    tokens_cached = []
    next_id = mx.argmax(logits_prefill[:, -1, :], axis=-1, keepdims=True)
    tokens_cached.append(next_id.item())
    for _ in range(n_generate - 1):
        logits = small_model(next_id, kv_cache=cache)
        next_id = mx.argmax(logits[:, -1, :], axis=-1, keepdims=True)
        tokens_cached.append(next_id.item())

    assert tokens_no_cache == tokens_cached  # ← must match exactly
```

This test passes. It's the single most important test in the inference pipeline — if the cached and uncached paths produce different tokens, something is fundamentally broken.

We also test that the full `Engine` (which adds the prefill/clone/sampling/tool-use layers on top) produces the same greedy output as `generate_naive`. Both pass.

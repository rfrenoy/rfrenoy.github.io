|-----------|-----------|
| AdamW | 1.43 |
| MuonAdamW | **0.03** |

Same model, same data, same number of steps. Muon reaches near-zero loss while AdamW is still struggling. Let's understand why.

## What Muon does

[Muon](https://kellerjordan.github.io/posts/muon/) — **M**oment**u**m **O**rthogonalized by **N**ewton-Schulz — is an optimizer designed specifically for matrix parameters. The core idea:

1. Compute the gradient (same as any optimizer)
2. Apply momentum (same as SGD with momentum)
3. **Orthogonalize** the update via polar decomposition

Step 3 is what makes Muon special. Instead of updating each weight independently (like AdamW), Muon replaces the gradient matrix $G$ with the nearest orthogonal matrix $U$, where $G = USV^T$ is the SVD.

### Why orthogonalize?

A gradient matrix $G \in \mathbb{R}^{M \times N}$ has singular values $\sigma_1 \geq \sigma_2 \geq \ldots \geq \sigma_{\min(M,N)}$. The problem: these singular values can vary by orders of magnitude. The largest singular values dominate the update, while the smallest directions get almost no gradient signal.

Orthogonalization replaces all singular values with 1:

$$G = U \text{diag}(\sigma_1, \sigma_2, \ldots) V^T \quad\longrightarrow\quad U \text{diag}(1, 1, \ldots) V^T = UV^T$$

Every direction gets equal update magnitude. This is a much better-conditioned update for training neural networks, where we want all weight matrix directions to learn at roughly the same rate.

Think of it as the matrix-valued analog of gradient normalization: just as dividing a vector gradient by its norm gives a unit-length update, replacing a matrix gradient with its polar factor gives an "equalized" update where no direction is preferred.

## Polar Express: fast approximate orthogonalization

Computing the exact SVD is $O(\min(M,N)^2 \cdot \max(M,N))$ — too expensive to run every training step. Muon originally used **Newton-Schulz iterations** to approximate the polar factor. nanochat has since switched to **Polar Express**, a newer iterative method with better convergence properties.

The algorithm: given a matrix $X$, iterate 5 times:

$$A = X X^T, \quad B = bA + cA^2, \quad X \leftarrow aX + BX$$

where $(a, b, c)$ are pre-computed coefficients optimized for convergence. After 5 iterations, $X$ is approximately orthogonal ($XX^T \approx I$). The whole thing runs in **bfloat16** for speed.

The result isn't exactly $UV^T$ — it's more like $US'V^T$ where the singular values $S'_{ii} \sim \text{Uniform}(0.5, 1.5)$. But this is close enough: the singular values are equalized to within a factor of 3, which empirically works just as well as exact orthogonalization.

Key differences:
- PyTorch uses `@torch.compile` to fuse the iteration into a single kernel. MLX relies on lazy evaluation to achieve similar fusion.
- PyTorch has `.mT` for the last-two-dims transpose; MLX uses `mx.swapaxes(X, -2, -1)`.
- PyTorch does in-place division (`/=`); MLX arrays are immutable, so we reassign.

## The full Muon pipeline

Polar Express is just step 3 of 5 in a Muon optimizer step:

1. **Nesterov momentum**: smooth the gradient using exponential moving average with Nesterov lookahead
2. **Polar Express**: orthogonalize the smoothed gradient
3. **NorMuon variance reduction**: normalize per-neuron update scales (the orthogonalized output can have non-uniform scales across neurons)
4. **Cautious weight decay**: only decay weights where the gradient and parameter agree in sign
5. **Parameter update**: apply the processed gradient

In nanochat's PyTorch version, all 5 steps are fused into a single `@torch.compile` kernel that operates on **stacked** parameter tensors — all matrix params of the same shape are stacked into one big tensor for efficient batch processing.

In our MLX version, we process parameters one at a time but rely on MLX's lazy evaluation to build one large computation graph that gets evaluated all at once:

## MuonAdamW: the combined optimizer

Not all parameters should use Muon. Embeddings, the output head, and scalar parameters (like the residual stream lambdas) are better served by AdamW. nanochat groups parameters by type:

| Parameter | Optimizer | Why |
|-----------|-----------|-----|
| Transformer matrices (Q, K, V, projections, MLP) | Muon | 2D matrices benefit from orthogonalized updates |
| Token embeddings (`wte`) | AdamW | Embedding lookups aren't matrix multiplications |
| Output head (`lm_head`) | AdamW | Tied to vocabulary size, different optimization dynamics |
| Value embeddings | AdamW | Same as token embeddings |
| `resid_lambdas`, `x0_lambdas` | AdamW | 1D scalars, Muon doesn't apply |

Each group also gets its own learning rate, betas, and weight decay — carefully tuned hyperparameters from nanochat's scaling experiments.

Our `MuonAdamW` class automatically classifies parameters by walking the model's parameter tree:

```python
optimizer = MuonAdamW(model, lr_schedule=schedule, matrix_lr=0.02, weight_decay=0.0)
# ...
loss, grads = loss_and_grad_fn(model, x, y)
optimizer.step(model, grads)
mx.eval(model.parameters())
```

## What we dropped

nanochat's optimizer is heavily optimized for multi-GPU training:

- **Stacked parameter processing**: groups all same-shape matrices into a single tensor for batch Polar Express
- **`@torch.compile` fusion**: the entire momentum → orthogonalize → update pipeline compiles to a single CUDA kernel
- **0-D CPU tensor trick**: hyperparameters passed as tensors to avoid recompilation when LR changes
- **`DistMuonAdamW`**: distributed version with overlapped `reduce_scatter`/`all_gather` and ZeRO-2 style state sharding

We don't need any of this:
- No multi-GPU → no distributed communication
- No `torch.compile` → MLX's lazy eval handles fusion
- No 0-D tensor trick → no compilation to avoid
- Processing params one at a time is fast enough on Apple Silicon for the model sizes we're targeting

Our optimizer is ~250 lines vs nanochat's ~400. Same algorithm, simpler wrapper.

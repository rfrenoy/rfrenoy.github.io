In the previous posts we built a GPT model ([post 1](/blog/01-gpt-model/)) and an efficient inference engine ([post 2](/blog/02-kv-cache-inference/)). But the model's weights are random — it produces gibberish. Time to fix that.

Training a neural network means repeatedly:
1. Feed data through the model (forward pass)
2. Measure how wrong the predictions are (loss)
3. Compute which direction to adjust each weight (gradients)
4. Nudge the weights in that direction (optimizer step)

The interesting part for this post is step 3 — computing gradients — because PyTorch and MLX do it in fundamentally different ways. This difference reshapes the entire training loop.

## Two philosophies of automatic differentiation

### PyTorch: the tape recorder

PyTorch records every operation during the forward pass onto a **computation graph** (sometimes called a "tape"). When you call `loss.backward()`, it walks this graph in reverse, computing gradients for every parameter along the way. The gradients accumulate silently in `.grad` attributes on each parameter tensor.

```python
# PyTorch training step
loss = model(x, y)          # 1. forward (records computation graph)
loss.backward()              # 2. backward (walks graph, fills .grad)
optimizer.step()             # 3. update params using .grad
model.zero_grad()            # 4. clear .grad for next step
```

This is *imperative* — side effects everywhere. The graph is global state. Gradients live on tensors. You must remember to zero them.

### MLX: the function transformer

MLX takes a *functional* approach inspired by [JAX](https://github.com/google/jax). Instead of recording a tape, you **transform** your loss function into a new function that also returns gradients:

```python
# MLX training step
def loss_fn(model, x, y):
    return model(x, targets=y)

loss_and_grad_fn = nn.value_and_grad(model, loss_fn)  # transform once

loss, grads = loss_and_grad_fn(model, x, y)  # 1+2. forward + backward in one call
optimizer.update(model, grads)                # 3. update params
mx.eval(model.parameters(), optimizer.state)  # 4. force computation
```

There is no tape, no `.grad` attributes, no `zero_grad()`. The `grads` variable is a plain Python dict tree that mirrors the structure of `model.parameters()` — you can inspect it, average it, clip it, or throw it away. It's just data.

### Why does this matter?

The functional approach has three practical consequences:

**No global state.** Gradients aren't hidden on parameter tensors — they're explicit return values. You always know where they are and what they contain.

**No zero_grad footgun.** In PyTorch, forgetting `model.zero_grad()` means gradients from the previous step leak into the current one. MLX returns fresh gradients each call.

**Lazy evaluation.** Nothing computes until `mx.eval(...)`. The entire forward-backward-update sequence is recorded as a computation graph and executed in one fused pass. This is MLX's performance trick: by seeing the whole pipeline at once, it can optimize memory allocation and kernel fusion across the forward and backward passes.

## Learning rate schedules

nanochat uses a 3-phase learning rate schedule:

1. **Linear warmup** from 0 to peak LR over 40 steps
2. **Constant** at peak LR
3. **Linear warmdown** to 5% of peak LR over the final 65% of training

$$
\eta(t) = \begin{cases}
\eta_{\text{peak}} \cdot \frac{t}{T_{\text{warmup}}} & t < T_{\text{warmup}} \\
\eta_{\text{peak}} & T_{\text{warmup}} \leq t \leq T - T_{\text{warmdown}} \\
\eta_{\text{peak}} \cdot \left(\alpha + (1-\alpha)\frac{T - t}{T_{\text{warmdown}}}\right) & t > T - T_{\text{warmdown}}
\end{cases}
$$

where $\alpha$ = `final_lr_frac` (typically 0.05).

In PyTorch, nanochat implements this as a manual `get_lr_multiplier(step)` function and loops over `optimizer.param_groups` each step to set the LR. In MLX, we build the schedule declaratively using MLX's built-in primitives:

MLX's `join_schedules` stitches multiple schedules together at specified boundaries. The optimizer calls the schedule internally on each `update()` call — no manual loop needed. The schedule is a callable that maps step count → learning rate, and the optimizer tracks its own step count.

## Gradient accumulation: implicit vs. explicit

When a single batch doesn't fit in memory (or when you want a larger effective batch size), you process multiple **micro-batches** and average their gradients before updating weights. This is called gradient accumulation.

In PyTorch, it's almost invisible — `.backward()` *adds* to `.grad` by default:

```python
# PyTorch: gradients accumulate automatically
for micro_step in range(4):
    loss = model(x, y) / 4   # scale loss by 1/N
    loss.backward()           # adds to .grad (doesn't overwrite!)
optimizer.step()
model.zero_grad()
```

In MLX, each `value_and_grad` call returns independent gradients. We have to sum them ourselves:

```python
def accumulate_gradients(loss_and_grad_fn, model, batches):
    total_loss = mx.array(0.0)
    total_grads = None

    for x, y in batches:
        loss, grads = loss_and_grad_fn(model, x, y)
        total_loss = total_loss + loss
        if total_grads is None:
            total_grads = grads
        else:
            total_grads = tree_map(lambda a, b: a + b, total_grads, grads)

    return total_loss / len(batches), tree_map(lambda g: g / len(batches), total_grads)
```

The `tree_map` function walks the nested gradient dict and applies a function to every leaf array — like a recursive `map` for trees. It's from `mlx.utils`, and it's one of the most commonly used utilities in functional ML frameworks.

This is more explicit than PyTorch's implicit accumulation, but it has a nice property: you always see exactly what's happening to your gradients. No hidden state, no surprises.

## The `mx.eval` question

The most common mistake when starting with MLX is forgetting `mx.eval`. Let's understand why it matters.

MLX is **lazy** — when you write `y = mx.matmul(a, b)`, no computation happens. MLX records the operation in a graph and returns a placeholder. Actual computation is deferred until someone calls `mx.eval(y)` or needs the concrete value (e.g., `y.item()`).

In the training loop, this means the entire forward pass, backward pass, and optimizer update are recorded as one giant computation graph. Nothing runs until:

```python
mx.eval(model.parameters(), optimizer.state, loss)
```

This single `eval` call triggers everything — forward, backward, weight update — in one optimized pass. MLX can see the full picture and fuse operations, reuse memory buffers, and schedule GPU work efficiently.

But if you forget it, computation piles up. After 100 steps without `eval`, you'd have 100 forward-backward graphs stacked in memory, and nothing would have actually computed. The program would appear fast (it's just recording, not computing!) and then crash with an out-of-memory error.

The rule: **call `mx.eval` at least once per training step**, passing everything that needs to be materialized.

To be clear: this is training on ~100K tokens of repeated literary quotes — it's memorization, not real language modeling. The model is learning to reproduce "To be or not to be" and "Call me Ishmael", not developing genuine language understanding. But it proves the training pipeline works: gradients flow, the optimizer updates weights, and the loss decreases.

## What's still missing for real training

This training loop works, but nanochat's full pipeline has several things we don't have yet:

**The Muon optimizer.** nanochat doesn't use plain AdamW for everything. It uses [Muon](https://kellerjordan.github.io/posts/muon/) — an optimizer that orthogonalizes gradients using polar decomposition — for the 2D matrix parameters, and AdamW only for embeddings and scalar parameters. Muon converges significantly faster than AdamW for transformer training. That's Phase 5.

**Real data.** Our tiny text dataset is 100K tokens of repeated sentences. nanochat trains on [FineWeb-Edu](https://huggingface.co/datasets/HuggingFaceFW/fineweb-edu-score-2) — billions of tokens of curated web text, fed through a sophisticated BOS-aligned best-fit dataloader. That's Phase 4.

**Scaling laws.** nanochat auto-computes the optimal batch size, training horizon, learning rate, and weight decay from the model depth using scaling law relationships (Chinchilla ratios, Power Lines batch size scaling, $T_{\text{epoch}}$ framework for weight decay). We just hard-code hyperparameters for now.

**Compilation.** nanochat wraps the model in `torch.compile(model, dynamic=False)` for a significant speedup. MLX has `mx.compile` which we haven't integrated yet.

But the core training loop — functional gradient computation, LR scheduling, gradient accumulation, optimizer updates — is complete and tested. In the next post, we'll build the data pipeline to feed it real text.
